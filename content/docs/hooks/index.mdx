OMX hooks extend Codex CLI at defined lifecycle boundaries. They inject developer context, activate skills, keep runtime state in sync, and continue long-running workflows without asking you to remember internal commands.

## Hook topics

| Topic | What it covers |
|---|---|

| [Codex-native Hooks](/docs/hooks/codex-native-hooks) | The concrete Codex hook integration points used by OMX |

## What hooks do in practice

- Detect workflow requests on `UserPromptSubmit`.
- Add guardrails before and after tool execution.
- Restore or continue active OMX modes like autopilot, ralph, ultraqa, and team.
- Persist `.omx/` state so long-running sessions survive interruptions.

If you are debugging why a skill activated, started, resumed, or refused to run, the hook docs are usually the fastest place to look.
