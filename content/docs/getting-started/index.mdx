oh-my-codex (OMX) is a workflow layer for [OpenAI Codex CLI](https://github.com/openai/codex). It keeps Codex as the execution engine and wraps it with role-specific agent prompts, reusable skills, and event-driven hooks so you can go from a vague idea to working code in one consistent loop. This page walks you through installing OMX, verifying the setup, and running a good first session.

## Prerequisites

- **Node.js 22+** (20+ is the minimum; 22 is recommended)
- **Codex CLI** installed and authenticated (`npm i -g @openai/codex`, then `codex login`)
- **macOS or Linux** — OMX is primarily designed and actively tuned for these platforms. Native Windows may break or behave inconsistently and currently receives less support.
- **tmux** (macOS/Linux) if you want the durable team runtime

## Install

<Steps>
<Step>
### Install Codex CLI

```bash
npm install -g @openai/codex
```

Then authenticate:

```bash
codex login
```
</Step>

<Step>
### Install OMX

```bash
npm install -g oh-my-codex
```
</Step>

<Step>
### Run setup

```bash
omx setup
```

`omx setup` wires the Codex-native hooks by writing `.codex/config.toml` (enables `codex_hooks = true`) and `.codex/hooks.json` (registers the OMX hook command). It also creates the `.omx/` project-state directory and configures the session prompt with OMX agent context.
</Step>
</Steps>

## A good first session

After install, check both boundaries:

```bash
omx doctor
codex login status
omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
```

`omx doctor` catches missing OMX files, hooks, and local runtime prerequisites. The `omx exec` smoke test proves the active Codex profile can actually authenticate and complete a real model call from your current shell and config.

Launch OMX the recommended way:

```bash
omx --madmax --high
```

If you explicitly want the interactive leader session inside tmux, use:

```bash
omx --tmux --madmax --high
```

Then try the canonical workflow:

```
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ralph "carry the approved plan to completion"
```

If the approved plan needs coordinated parallel execution, switch the last step to:

```
$team 3:executor "execute the approved plan in parallel"
```

Use `--madmax` intentionally: it bypasses Codex approvals and sandboxing. Use `$team` when the approved plan needs coordinated parallel work, or `$ralph` when one persistent owner should keep pushing to completion.

If you prefer a more hands-off, one-command workflow after learning the default path, [`$autopilot`](/docs/skills/workflow/autopilot) is still available.
